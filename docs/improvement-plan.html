Closing the gap between what the platform
promises and what the code enforces.

A four-track audit — security, agent handoffs, observability, and UI — of the multi-agent coordination platform. The contracts and architecture are strong; the work ahead is making the running system live up to them.

~11.4k LOC audited 4 tracks 32 findings threat model: trusted local operator 2026-07-03

State of the system

  1. The safety layer isn't on the path agents use. The MCP shell_exec / fs_read / fs_list tools bypass the policy engine, approval queue, and worktree isolation entirely — and the one place the pipeline is wired auto-approves every action marked "requires approval."
  2. Agent-to-agent handoff doesn't work with the real binaries. Codex/Hermes bridges can only emit WriteMemory + TaskDone — they can't initiate a handoff, and the bundle they'd pass is nearly empty. True multi-hop handoff is exercised only by a test fixture.
  3. Observability fragments the story. Every read re-parses the whole audit log (O(n), 17+ endpoints), the true actor of a handoff hides in an untyped payload, reads are never audited, and there's no single "what changed and why" view.
  4. The UI is 7 bespoke pages with no shared core. el() is redefined 8×, colors and time-formatters drift into visibly wrong output, keyboard access is broken everywhere, and 30-second polls wipe half-typed operator input.

Security & safety

Make the guarantees true

CLAUDE.md's "no tool executes without a PolicyDecision" and the README's "read-only lens" are both false as written. These are correctness bugs even under the benign model — and remote-exploitable the moment --host 0.0.0.0 is used.

A1P0MCP fs/shell tools bypass the entire policy engine

The three tools agents actually call read files, list directories, and run subprocesses directly — no ToolExecutor, no policy, no approval, no workspace confinement. Arbitrary command execution and arbitrary file read/write. The whole policy/ package is dead code on the primary surface.

operator/mcp/server.py:835–945
Fix
Route all three through ToolExecutor.invoke() with a real PolicyEngine + InvocationContext + workspace root, exactly like the builtin path. Delete the direct read_text/iterdir/create_subprocess_exec calls.
A2P0Web server has no auth, CSRF, or Origin check — and isn't read-only

The "lens" exposes state-changing endpoints including POST /api/conversations/{id}/run, which spawns real codex/hermes subprocesses. The event WebSocket accepts any Origin, so any site the operator visits can stream the full audit feed or trigger dispatches via CSRF.

web/server.py:26–99 · routes.py:731, 1679–1703
Fix
Enforce loopback-only bind; add a localhost-origin / token check on all /api/*; add an Origin/Host allowlist on the WebSocket handshake; require a non-forgeable header on every non-GET route.
A3P1Dispatched Codex runs sandbox-disabled inside a fake worktree

--dangerously-bypass-approvals-and-sandbox is set while the "worktree" is a plain mkdir, not git worktree add — and the docstring claims an isolation the flag explicitly removes.

mcp/dispatch.py:81–104, 202–225 · bridge/codex.py:111–117
Fix
Use a real git-worktree manager and keep Codex's sandbox on, or gate the bypass behind an explicit operator-confirmed unsafe mode.
A4P1"Requires approval" is silently auto-approved in every dispatch

auto_approve_resolver downgrades REQUIRE_APPROVAL rules (fs.write, shell.exec) to instant approval — no human in the loop, and the approval timeout is never enforced.

mcp/dispatch.py:153 · policy/approvals.py:17–19
Fix
Provide a real file-backed approval queue, or mark these ALLOW explicitly and stop advertising them as gated.
A5P2Secrets leak into the shared memory DB and child-process env

shell_exec auto-records raw argv (passwords, bearer tokens) into shared memory, surfaced to every agent and the UI; every subprocess inherits the daemon's full environment (env= never scrubbed).

mcp/server.py:909–938 · tools/builtin/shell.py:22–27
Fix
Redact secret-shaped tokens before recording; pass a scrubbed env= allowlist to every spawn.
A6P2fs builtins have no path validation of their own

Containment relies entirely on the policy layer, so any caller that skips the executor gets unrestricted traversal. _fs_write even mkdir(parents=True).

tools/builtin/fs.py:10–36
Fix
Enforce a configured sandbox root inside each handler (resolve + relative_to) as defense in depth.
A7P2install.sh is curl|bash off a mutable branch, unverified

Clones a mutable main, pipes a second remote script to sh, and runs uv sync (arbitrary build hooks) — no pinning, checksum, or signature.

install.sh:6–7, 54–81
Fix
Pin to a tag/commit, verify a checksum, prefer download-inspect-run over nested pipe-to-shell.
A8P2FTS5 MATCH on raw query → uncaught 500 / cheap DoS

Inputs like ", NEAR(, col: raise an uncaught OperationalError. (No SQL injection — all queries are parameterized.)

memory/durable.py:140–168
Fix
Wrap FTS parsing in try/except → clean 400, or quote the query into a phrase.
B

Agent-to-agent handoffs

Priority #1 · make the bundle real

The Handoff schema is rich — goal restatement, constraints, decisions, open questions, workspace state, tool cursor, budget. The failure is that almost nothing populates it, and receivers read almost none of it.

B1P0Real bridges cannot initiate a handoff at all

Codex/Hermes always build exactly [WriteMemory, TaskDone]; only the test ScriptedProcess emits RequestHandoff. The coordinator's if not handoff_out.to_agent: break terminates every real chain after hop 0.

bridge/codex.py:204–207 · bridge/base.py:220–225 · coordinator.py:107
Fix
Teach the bridges to parse a structured handoff request from the agent's final message (or a dedicated MCP tool) and translate it into RequestHandoff.
B2P0The bundle is nearly empty; receivers read only the goal

workspace_state is hardcoded None; decisions and open questions are always empty; budget carries the original amount; the tool cursor is empty. The digest reads session memory while agents write to durable memory, so it's usually empty and the goal collapses to task.goal.

bridge/base.py:150–179 · summarizer.py:43–89
Fix
Populate workspace_state; digest durable records; have receivers prepend constraints + decisions + open questions + expected output to the prompt.
B3P1"Synthesize turn" fabricates content in conversations

With decisions empty it falls to goal_restatement — which in the conversation path is the instruction prompt. The transcript records the instruction reflected back as the agent's reply, while the real response record is never reached.

conversation.py:390–418, 526–541
Fix
Reorder _extract_agent_reply to prefer the actual response record; never fall back to the goal when it equals the dispatched goal.
B4P1claude_code→codex fallback misattributes the speaker

The fallback fires on the name, not capability. In conversations codex produces the text but the turn is recorded as from_agent="claude_code" — the transcript lies about who spoke.

mcp/dispatch.py:322–356 · conversation.py:494–543
Fix
Propagate the effective agent into attribution; clearly re-label substituted turns.
B5P1Anonymous handoffs slip through chain-of-custody

from_agent is schema-required but optional on the MCP tools; the audit event records from_agent=None, and to_agent is logged as "auto" for capability-routed dispatches — which children then can't inherit.

mcp/server.py:207–215 · dispatch.py:388–421, 515–534
Fix
Make from_agent required on the tools (fail fast); record the resolved agent as to_agent, never "auto".
B6P1Routing is name-based; capability routing is dead code

CapabilityRouter.route has real logic, but required_capabilities is never plumbed from any MCP tool into task.inputs, so it always falls through to "first registered agent."

router.py:64–92 · dispatch.py:369–427
Fix
Add a required_capabilities parameter to the dispatch tools and thread it into task_inputs.
B7P1Worktrees leak on every task

Coordinator.submit creates a worktree and never removes it (success, failure, or exhaustion); dispatch scratch dirs are never GC'd. Unbounded disk growth and a bloating git worktree list.

coordinator.py:55, 129–142 · dispatch.py:96–103
Fix
Wrap submit in try/finally → worktree_mgr.remove; add a TTL sweep for dispatch scratch dirs.
B8P1The merge gate never merges or detects conflicts

It records events and the coordinator auto-accepts — no git merge, no conflict detection. Two tasks touching the same files are both declared "merged" while nothing is integrated.

merge_gate.py:29–89 · coordinator.py:129–139
Fix
Implement git merge --no-ff (or PR creation) with conflict → operator review; emit HANDOFF_ACCEPTED only on a clean merge.
B9P1Dispatch/task/session state dies on restart

_running, _tasks, _sessions are in-memory dicts; after an MCP restart, in-flight dispatch handles become permanently unresolvable — despite docstrings claiming "reconstructible from the audit log."

dispatch.py:138, 478–513 · task_manager.py:11–21
Fix
Rehydrate registries from the audit log on startup, or move dispatch state to durable storage.
B10P2budget_remaining is never decremented

The coordinator only calls budget.check(); tokens/dollars/approvals usage is never recorded, and the handoff carries the full original budget. Budget hand-off is cosmetic.

coordinator.py:105 · budget.py:31–74 · base.py:177
Fix
Record usage as tools/approvals fire; thread limit − used into build_handoff.
B11P2Inbox is poll-only, rescans the whole log, drops turns

O(total events) per agent per turn (the 10–30s latency). The since_ms filter uses <=, silently dropping same-millisecond turns; pending messages in a closed conversation are lost.

conversation.py:258–293
Fix
Ship the bridge-push protocol; interim, index turns by conversation and use a strict > cursor keyed on (ts, turn_id).
B12P2Fallback can route to a capability-mismatched agent

With no required_capabilities, find_fallback(required=None) returns the next registered agent regardless of what it can do; a failed edit_files task can "succeed" with a no-op.

coordinator.py:201–248 · router.py:97–112
Fix
Infer required capabilities from the failed agent's declared set when the task didn't specify them.
C

Observability & read-only audits

Priority #2 · who / what / why / order, in one place

Today, answering "what changed and why" means stitching across /debug, /agents, the chains drawer, and /memory — and every read re-parses the whole log.

C1P1The event model hides the story in an untyped payload

On HANDOFF_INITIATED the top-level agent_id is "exocortex" — the real actor is buried in payload["to_agent"]. No correlation_id, causation_id, or top-level parent_task_id; ordering is wall-clock only; no human-readable "why," and policy_decision.reason isn't even rendered.

contracts/event.py:61–76 · render.py:37–49
Fix
Promote to typed Event columns: actor, parent_task_id, caused_by_event_id, optional reason. This one change fixes WHO/WHY/ORDER and makes chains robust to sloppy agents.
C2P1Every read re-parses the entire audit log

read_all() reads and Pydantic-validates the whole file; 17+ endpoints call it fresh, and the dashboard polls several on 10–30s timers — re-parsing the log many times per minute. O(events) per request in IO and validation.

audit.py:33–42 · routes.py (17 call sites)
Fix
Maintain a shared, append-aware in-process event cache (tail from last byte offset — events.py already does this) plus a task_id → offset index used by all read endpoints.
C4P1Reads are never audited

memory_search/get/list emit no event, so "what did the agent consult before deciding X" is unanswerable. With no derived_from links, the decision→justification chain can't be shown at all.

handlers.py:446–582 · event.py:23–27
Fix
Add a MEMORY_READ event kind emitted by the read handlers.
C5P1CLI and web renderers diverge; trace is barely readable

The nice per-kind formatters live only in routes.py. precog trace dumps raw k=v truncated at 80 chars, shows only %H:%M:%S (no date — a task spanning midnight is unreadable), and never renders reason. Many kinds render a blank summary.

render.py:37–49 · routes.py:163–197
Fix
Lift one EventKind → human sentence formatter shared by CLI + web; add the missing kinds; render reason/policy_decision; add a full date.
C6P1No "trace everything" view; can't follow a chain

/debug is failure-only (hardcoded 4 kinds), with no free-text search and no export. precog trace matches a single task_id, so following work across a handoff (a new child task id) means finding it by hand and re-running.

debug.js:56–62 · cli.py:158–162
Fix
Unify /debug into an all-kinds, text-searchable, exportable event browser; add precog trace --follow-chain that walks child tasks.
C3P2No log rotation, compaction, or index

The code anticipates a rotated log but nothing ever rotates it; orphan events (missing TASK_CREATED) are silently dropped from reconstruction.

config.py:19 · audit.py · recall.py:165–167
Fix
Periodic snapshot/rotation with an explicit "rotated" marker so recall/trace know pre-rotation work is archived, not lost.
C7P2Chain/swimlane rendering loses data silently

The agent color map is hardcoded and duplicated (drifts between files), tasks with an underivable owner are skipped and vanish from the swimlane, and summarizePayload reads p.tool while the server writes tool_name. Also, tracing.py is a no-op OTel shim — spans emit nothing.

chains.js:29–35, 306–307 · dashboard.js:708–718 · tracing.py:14–30
Fix
Data-drive and dedupe the color map; render a placeholder lane for unknown-agent tasks; align field names; wire real OTel or drop the shim.

Web UI

Priority #3 · a shared foundation

Seven internally-coherent pages with no shared front-end core. The palette and token discipline are good; the problems are structural duplication, broken keyboard access, and re-renders that fight the operator.

D1P1No shared module — duplication that ships wrong output

el() is redefined in 8 files, the relative-time formatter 4× with different granularity (one event reads "45d ago" on one page, "1mo ago" on another), AGENT_COLORS 3× with a mismatched fallback (same agent, different color per page), and highlightJson emits .k vs .json-k so the widget can't share CSS.

static/*.js (all)
Fix
Extract static/common.js (el, colors, time, highlightJson, connectWs, fetchJSON, a shared drawer controller) + a shared header with one canonical URL set.
D2P1Keyboard accessibility is broken platform-wide

outline:0 appears 10× with no :focus-visible anywhere. Core interactions are clickable <div>s with no tabindex/role/key handler. The constellation canvas and swimlane bars are mouse-only and invisible to screen readers; the swimlane hint says "esc closes" but no handler exists; drawers lack role="dialog" and focus management.

app.css (10 sites) · agents.js · constellation.js · chains.js
Fix
Global :focus-visible; convert clickable rows to real <button>s; give the canvas + swimlane a focusable text-equivalent; shared drawer with role="dialog" + focus trap/restore.
D3P130s polls clobber in-progress operator input

The profile questions poll wipes a half-typed answer (and focus) every 30s — destroying exactly the interaction the page exists for. The agents timeline rebuild destroys the open "why" drawer mid-read; a 500 leaves the profile page stuck on "loading…" forever.

profile.js:662, 952 · agents.js:420, 706–715
Fix
Diff-render by stable id (skip re-render while a textarea has focus/content); set the loaded flag on every error path; render any non-ok as a distinct error state.
D4P1The constellation is untrustworthy and expensive

Edges labeled "semantic ≥ 0.78" are actually layout-proximity edges; a 30-day hard cap permanently hides older records from a durable store; there's no loading/empty/error state (plus a crash when points is missing). It rebuilds 12 label divs and re-uploads the entire point buffer every frame, even idle.

constellation.js:527–563, 98, 1666–1725 · memory.html:70, 92
Fix
Relabel or drop the fake semantic edges; add an unbounded age option; guard null/empty/error/loading; render-on-demand (build labels once, dirty-flag uploads).
D5P1Live-update gaps — no backfill, debug has no socket

/debug — where you most want live pushes — has no WebSocket (up to 30s latency). On reconnect there's no backfill, and the agents timeline updates only via WS pushes with no reconcile, so a dropped socket silently loses history. A running conversation is a synchronous POST /run that can block up to 900s.

debug.js:571 · agents.js:736 · routes.py:1679–1703
Fix
Wire the shared connectWs into debug; re-run the primary fetch on WS open to reconcile; make /run return a job id and push progress over the WS.

What the audit did not find

Checked, clean — and patterns worth promoting
  • No SQL injection — all durable.py queries use ? placeholders.
  • No unsafe deserialization — only json + Pydantic; no pickle/yaml.load/eval.
  • No shell=True — both shell paths use create_subprocess_exec(*argv); the risk is unrestricted argv/cwd, not metacharacter injection.
  • Default bind is 127.0.0.1 (safe) — the risk is the absence of defense-in-depth if that's changed.
  • Good patterns to promote: the dashboard/agents/tasks WebSocket reconnect uses correct exponential backoff, and debug.js's fetchJSON wrapper is the cleanest in the codebase — both belong in the shared core.
